# CloudDB Hub & Node Documentation

## 1. Component Model

CloudDB uses a Hub-and-Node model.

- Hub:
  - PHP application generated by `install-db-hub.sh`
  - stores metadata in SQLite
  - manages tenants, packages, Nodes, health checks, and provisioning requests
- Node:
  - MariaDB server plus a lightweight PHP agent generated by `install-db-node.sh`
  - executes create, delete, backup, whitelist, stats, and password rotation actions

The Hub never stores tenant database data. It stores only platform metadata and the mapping between tenants and Nodes.

## 2. Important Files And Paths

Hub:

- App root: `/var/www/db-hub`
- Main app: `/var/www/db-hub/index.php`
- SQLite DB: `/var/www/db-hub/hub_v5.sqlite`
- Install summary: `/root/db-hub-install-summary.txt`

Node:

- Agent root: `/var/www/db-agent`
- Agent entrypoint: `/var/www/db-agent/agent.php`
- Apache alias for agent: `/agent-api`
- phpMyAdmin alias: value chosen during install, usually `/phpmyadmin`
- Backup directory: `/var/backups/mariadb`
- Backup script: `/usr/local/sbin/db-platform-backup.sh`
- Backup credential file: `/etc/clouddb/backup-mysql.cnf`
- Backup log: `/var/log/clouddb-backup.log`
- Install summary: `/root/db-node-install-summary.txt`

## 3. Credential Sources

### Hub Summary

File: `/root/db-hub-install-summary.txt`

Fields:

- `Access`: Hub login URL.
- `TLS`: whether automatic HTTPS succeeded.
- `TLS Command`: manual fallback command for Certbot if automatic HTTPS did not complete.
- `Admin Identity`: initial Hub admin username.
- `Access Key`: initial Hub admin password.

When needed:

- Use `Admin Identity` and `Access Key` for the first Hub admin login.
- If `TLS` is not fully enabled, run the saved `TLS Command` after confirming DNS and firewall readiness.

### Node Summary

File: `/root/db-node-install-summary.txt`

Fields:

- `Agent Key`
- `Backup Key`
- `Provisioner`
- `Agent Path Restriction`
- `phpMyAdmin Alias`
- `Backup Directory`
- `Backup Schedule`
- `Backup Log`

When needed:

- `Agent Key`: required when adding the Node in the Hub UI.
- `Backup Key`: required when decrypting and restoring encrypted backups.
- `Provisioner`: internal service credential for the Node agent. This is not a tenant credential.
- `Agent Path Restriction`: useful when diagnosing failed Hub-to-Node health checks.
- `phpMyAdmin Alias`: needed when filling the Hub Node form or testing direct Node access.
- phpMyAdmin configuration storage is now bootstrapped automatically by the Node installer using the `phpmyadmin` database and a local control user.

## 4. Registering A Node In The Hub

The Hub Node form expects:

- `Name`: display label only.
- `Database Host / IP`: the MariaDB host written into tenant `.env` files.
- `Public Endpoint`: the Node base URL used for health checks and agent calls.
- `Agent Access Token`: the `Agent Key` from the Node summary.
- `phpMyAdmin Alias`: the alias from the Node summary.

Rules:

- `Public Endpoint` must be the base URL only.
- Do not append `/agent-api/agent.php`.
- If the MariaDB service is on a non-default port, enter `host:port` in `Database Host / IP`.

## 5. Tenant Provisioning Flow

Provisioning can start from either side:

- Admin provisions for a tenant from the admin tenant table.
- Tenant provisions directly from the client dashboard.

Provisioning sequence:

1. Hub checks tenant package limits.
2. Hub syncs current tenant usage.
3. Hub picks the healthiest available Node based on agent `stats`.
4. Hub requests database creation from that Node.
5. Node creates the database and DB user.
6. Hub stores the tenant-to-Node mapping in SQLite.
7. Hub prepares the downloadable `.env`.

The generated `.env` includes:

- `DB_HOST`
- `DB_PORT`
- `DB_DATABASE`
- `DB_USERNAME`
- `DB_PASSWORD`

`DB_HOST` is taken from the Node `Database Host / IP` field. `DB_PORT` defaults to `3306` unless a `host:port` value is configured there.

## 6. Admin Features

The current admin UI supports:

- Node add, edit, delete
- full-node backup trigger
- tenant create, edit, delete
- tenant provision trigger
- tenant backup trigger
- latest generated `.env` download

Safety behavior:

- Nodes with attached tenant databases cannot be deleted.
- Tenants with attached databases cannot be deleted.
- Full-node backup buttons are disabled when the Node is not currently considered healthy by the Hub.

## 7. Client Features

The current client UI supports:

- login
- self-service database provisioning
- `.env` download
- phpMyAdmin jump link
- IP whitelist updates
- per-database backup trigger

## 8. Backup Model

### Automatic Backups

Installed on every Node:

- cron schedule: `0 2 * * *`
- command: `/usr/local/sbin/db-platform-backup.sh`
- output: encrypted `.sql.gz.enc` files in `/var/backups/mariadb`
- credentials: `/etc/clouddb/backup-mysql.cnf`
- log file: `/var/log/clouddb-backup.log`

### Manual Backups From The Hub

Hub actions queue Node backup jobs asynchronously:

- client dashboard backup button:
  - one tenant database
- admin tenant backup button:
  - every database owned by that tenant
- admin Node backup button:
  - full-node dump

The Node backup script:

1. runs `mariadb-dump` or falls back to `mysqldump`
2. compresses with `gzip`
3. encrypts with OpenSSL AES-256-CBC using the Node `Backup Key`
4. connects with the dedicated provisioner credential file instead of root socket auth
5. optionally syncs `/var/backups/mariadb` with `rclone`
6. deletes encrypted files older than 30 days

## 9. Restoring A Backup

Restore prerequisites:

- encrypted backup file
- matching `Backup Key`
- MariaDB installed and running on the destination server

Restore command:

```bash
openssl enc -d -aes-256-cbc -pbkdf2 -pass pass:YOUR_BACKUP_KEY -in /var/backups/mariadb/your-backup.sql.gz.enc | gunzip | mariadb
```

Recommended disaster-recovery flow:

1. Install or rebuild the destination Node.
2. Copy the encrypted backup file to that server.
3. Retrieve the original `Backup Key` from `/root/db-node-install-summary.txt` or your secrets store.
4. Run the restore command.
5. Re-register or update the Node in the Hub if the server identity changed.
6. Verify tenant connectivity and phpMyAdmin access.

## 10. phpMyAdmin Visibility

The Node installer now creates phpMyAdmin configuration storage automatically and writes a phpMyAdmin config fragment that hides:

- `information_schema`
- `performance_schema`
- `mysql`
- `sys`
- `phpmyadmin`

This improves the tenant UI in phpMyAdmin. It does not remove those internal MariaDB schemas from the database engine itself.

## 11. Health Checks

Hub health monitoring:

- watchdog route: `index.php?action=watchdog`
- schedule: every 10 minutes from cron
- result:
  - updates `last_seen` when the Node responds
  - sends an alert email when the Node is unreachable

## 12. Operational Caveats

- Hub TLS is now attempted automatically during install when a valid FQDN and admin email are supplied. If issuance fails, the Hub summary includes the exact fallback Certbot command to run manually.
- Node TLS is not automatically completed by the current installer; if you use `https://` in the Hub Node form, make sure the Node actually serves TLS.
- The agent API still uses a query-string API key. Protect the Node with network controls and the `Agent Path Restriction`.
- phpMyAdmin schema hiding is a UI-level measure. It is not a substitute for MariaDB privilege design.

## 13. Updating The Generated App

The actual runtime PHP for both sides is generated from heredocs inside:

- `install-db-hub.sh`
- `install-db-node.sh`

If you change platform behavior, update the installer heredocs first, then redeploy or rerun the installer on the target server.

## 14. Uninstalling The Platform

Supported uninstall targets:

- `sudo ./uninstall-db-platform.sh hub`
- `sudo ./uninstall-db-platform.sh node`
- `sudo ./uninstall-db-platform.sh all`

Optional flags and environment variables:

- `PURGE_PACKAGES=yes`: also purge related OS packages
- `--remove-repo` or `REMOVE_REPO=yes`: also remove the installer checkout directory after uninstall

Important behavior:

- By default, the uninstaller removes the deployed Hub and Node runtime only.
- The installer checkout removal is conservative and only runs when the script can confirm the directory still looks like the CloudDB repo.
- Removing the repo checkout is useful on a server where you cloned the project only for install and no longer need the source tree afterward.
